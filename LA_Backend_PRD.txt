1. Purpose & Scope
This document defines the complete product requirements for the LinkinAgency Backend API — the server-side service that receives LinkedIn HTML captures from the Chrome extension, parses them into structured JSON using Cheerio, stores the JSON in MongoDB, and archives the raw HTML file to Cloudinary.


Every section is written for an AI coding agent to implement directly. No requirement should require interpretation. Ambiguous cases are resolved in Section 11 (Open Questions).

1.1 Responsibilities
	•	Accept multipart/form-data POST requests from the Chrome extension
	•	Validate the incoming payload — reject malformed or incomplete requests early
	•	Upload the raw HTML file to Cloudinary under a structured folder path
	•	Run the Cheerio parser against the HTML and produce a structured JSON object
	•	Write a capture record to MongoDB containing: metadata, parsed JSON, Cloudinary file URL
	•	Return a clear JSON response to the extension indicating success or failure

1.2 Out of Scope
	•	Authentication beyond API key header validation (OAuth deferred to V2)
	•	Real-time webhooks or push notifications
	•	Analytics computation or scoring (that is the AI pipeline's job, downstream)
	•	Admin dashboard UI

5.3 Processing Pipeline (in order)
	•	Validate API key from x-api-key header — reject 401 if missing or invalid
	•	Receive multipart request via Multer — store file in memory buffer (not disk)
	•	Validate all required fields via Zod schema — reject 400 with field errors if invalid
	•	Validate file MIME type is text/html and size ≤ 10MB — reject 422 if not
	•	Upload HTML buffer to Cloudinary — get back secure URL and public_id
	•	Run Cheerio parser on HTML buffer — produce parsedData JSON object
	•	Build Mongoose Capture document from all fields + cloudinaryUrl + parsedData
	•	Save Capture document to MongoDB
	•	Return 201 response with capture _id, cloudinaryUrl, parsedData summary

6. Cloudinary Integration


Cloudinary is chosen over ImageKit for this use case. Reason: Cloudinary's Node SDK supports raw file upload (not just images) with a simple upload_stream API, has a generous free tier (25GB storage, 25GB monthly bandwidth), and returns stable permanent HTTPS URLs immediately. ImageKit requires additional raw media configuration that adds setup friction.

6.1 Folder Structure on Cloudinary
All uploads must follow this path convention. Never upload to the root.

linkinagency/
└── clients/
    └── {clientId}/
        └── {pageType}/
            └── {YYYY-MM-DD}/
                └── {capturedAt-timestamp}.html

Example:
linkinagency/clients/client_sapna_nibsaiya/analytics_posts_impressions_90d/2026-02-22/1708608600000.html

6.2 Upload Configuration

6.3 Error Handling for Cloudinary
	•	If Cloudinary upload fails for any reason — return 500 with code CLOUDINARY_UPLOAD_FAILED
	•	Do NOT proceed to save the MongoDB record if Cloudinary fails
	•	Log the full Cloudinary error object to the logger
	•	Include the Cloudinary error message in the server log but NOT in the API response


7. Parser Service — Cheerio
The parser service receives the HTML buffer as a string and returns a structured JSON object. It must not throw — all errors must be caught internally and returned as a partial object with a parse_error field.

7.1 Parser Input
	•	Input: UTF-8 string of the HTML file contents
	•	Input: pageType string (to select the correct parsing strategy)

7.2 Parser Output — Shape
The parser always returns an object with this top-level shape:

{
  pageType:    String,          // echoed from input
  parsedAt:    ISO String,       // timestamp of parse execution
  parse_error: String | null,    // null if successful, error message if partial
  data:        Object            // page-type-specific payload (see 7.3)
}

7.4 Parser Rules
	•	The parser must never throw an uncaught exception
	•	If a selector finds nothing, return null for that field — do not return undefined
	•	If the entire parse fails, return the top-level object with parse_error set and data as an empty object {}
	•	Numbers must be returned as JavaScript Numbers (not strings) — strip commas before parsing
	•	Percentages are returned as the numeric value only — e.g. 72.6, not '72.6%'
	•	Text fields are trimmed and whitespace-collapsed before storage
	•	Post text is capped at 500 characters
	•	Demographics arrays are capped at 10 items each
	•	topPosts array is capped at 20 items

9. Authentication
V1 uses a static API key passed in the x-api-key request header. This is sufficient for an internal tool used by the LinkinAgency team only.

Implementation
	•	API_KEY value is set in the .env file
	•	The apiKey.js middleware compares req.headers['x-api-key'] against process.env.API_KEY
	•	If missing or mismatch — return 401 immediately, do not proceed
	•	Apply this middleware to all /api/* routes
	•	The /health endpoint is public — no API key required


Never log the API key value. If the key is compromised, rotate it in .env and restart the server. V2 will add per-client OAuth tokens.

11. Cloudinary Setup — README
This section is the complete Cloudinary onboarding guide for a new developer setting up the backend for the first time.

Step 1 — Create a Free Cloudinary Account
	•	Go to cloudinary.com/users/register/free
	•	Sign up with Google or email
	•	Choose 'Programmable Media' as your use case
	•	After signup you land on the Dashboard

Step 2 — Get Your Credentials
	•	On the Cloudinary Dashboard, look for the 'API Keys' section
	•	You will see three values: Cloud Name, API Key, API Secret
	•	Copy all three into your Backend/.env file under the CLOUDINARY_* keys


The API Secret is shown only once in some flows. Copy it immediately and store it in your .env. Never paste it in Slack or commit it to git.

Step 3 — Configure Upload Preset (Optional but Recommended)
	•	In the Cloudinary Dashboard → Settings → Upload → Upload Presets
	•	Click 'Add upload preset'
	•	Set Signing Mode to 'Signed'
	•	Set Folder to 'linkinagency'
	•	Set Resource Type to 'Raw'
	•	Save the preset name — you can pass it as the upload_preset option in the SDK

Step 4 — Install the Cloudinary SDK
npm install cloudinary

Step 5 — Initialise in config/cloudinary.js
const { v2: cloudinary } = require('cloudinary');

cloudinary.config({
  cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
  api_key:    process.env.CLOUDINARY_API_KEY,
  api_secret: process.env.CLOUDINARY_API_SECRET,
  secure:     true,
});

module.exports = cloudinary;

Step 6 — Upload an HTML File (Stream Pattern)
Multer stores the file in memory as a Buffer. Use an upload stream to send it to Cloudinary without writing to disk:

const { Readable } = require('stream');

function uploadHtmlToCloudinary(buffer, options) {
  return new Promise((resolve, reject) => {
    const stream = cloudinary.uploader.upload_stream(
      { resource_type: 'raw', format: 'html', ...options },
      (error, result) => {
        if (error) reject(error);
        else resolve(result);
      }
    );
    Readable.from(buffer).pipe(stream);
  });
}

Step 7 — Verify Upload
	•	After running the server and sending a test capture, open Cloudinary Dashboard → Media Library
	•	Navigate to linkinagency/clients/... and confirm the HTML file is stored there
	•	Copy the Secure URL and paste it in a browser — you should see the raw HTML


12. Error Handling — Global Rules
	•	All route handlers must be wrapped in try/catch or use an async wrapper middleware
	•	Unhandled errors bubble to the global errorHandler.js middleware
	•	errorHandler always returns JSON — never raw Express HTML error pages
	•	All errors are logged with: timestamp, route, error code, error message, stack trace (dev only)
	•	5xx errors must not expose internal implementation details in the response body
	•	4xx errors must include actionable detail — which field failed, what was expected


13. Acceptance Criteria
The backend is complete when ALL of the following pass:

	•	POST /api/capture with valid multipart payload returns 201 and a captureId
	•	The HTML file appears in the correct Cloudinary folder path (as per Section 6.1)
	•	The Capture document in MongoDB contains cloudinaryUrl, parsedData, and parseSuccess: true
	•	POST /api/capture with missing clientId returns 400 with field-level error details
	•	POST /api/capture with wrong x-api-key returns 401
	•	POST /api/capture with a 12MB file returns 413
	•	POST /api/capture with a .pdf file returns 422
	•	GET /api/capture/client/{clientId} returns only records for that client, excluding deleted ones
	•	DELETE /api/capture/:id sets deleted: true but does not remove the document from MongoDB
	•	GET /health returns 200 with { status: 'ok', uptime: Number }
	•	If the Cheerio parser cannot extract the main metric, parsedData.parse_error is non-null and parseSuccess is false — but the record is still saved
	•	Sending two identical captures (same clientId, pageType, capturedAt) stores two separate records — no deduplication in V1
